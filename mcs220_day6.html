🔄 1. The Servlet Life Cycle

init() → service() → destroy() — controlled by the Web Container

📖 Explain

A Servlet's life is strictly managed by the Web Container (e.g., Apache Tomcat). Three core methods define its existence:

  • init(): Called exactly once when the servlet is first loaded. Used for one-time setup (DB connections, configuration).
  • service(): Called for every client request. Handles request/response logic (doGet/doPost in HttpServlet).
  • destroy(): Called exactly once before servlet is unloaded (server shutdown). Cleanup resources.
🧪 Lab Insight (MCSL-222): First hit after starting Tomcat triggers init() (slight delay). Rapid refreshes only call service(). Server stop triggers destroy().

🍪 2. Session Management & Cookies

Overcoming HTTP statelessness — session cookies & persistent cookies

📖 Explain

HTTP is stateless — each request is independent. Session management creates "memory" across requests. Cookies are small text files stored by the browser and sent with every request.

  • Non-Persistent (Session Cookie): Lives in browser memory, deleted when browser closes. Used for login sessions.
  • Persistent Cookie: Stored on hard drive, has expiry date. Used for "Remember Me" features.
🛒 Example: Shopping cart uses session cookie to track items. "Remember Me" checkbox creates persistent cookie.

📝 Exam-Style Questions & Answers (Check Your Progress)

Q1. Explain the Servlet Life Cycle. (Highly repeated PYQ)
Answer: The Servlet Life Cycle consists of three phases managed by the Web Container: 1. Initialization (init()): Called once when servlet is instantiated — used for one-time setup. 2. Servicing (service()): Called for each client request — processes request and generates response. In HttpServlet, service() delegates to doGet/doPost. 3. Destruction (destroy()): Called once before servlet is unloaded — releases resources like database connections.
Q2. Compare the Generic Servlet with HTTP Servlet.
Answer: GenericServlet: Protocol-independent, abstract class implementing Servlet interface. Provides basic lifecycle methods; service() is abstract. HttpServlet: Extends GenericServlet, designed specifically for HTTP protocol. Overrides service() to route requests to doGet(), doPost(), doPut(), etc. Used for web applications.
Q3. Why is Session Management required in web applications?
Answer: Session management is required because the HTTP protocol is stateless. Without it, the server cannot identify returning users across multiple requests. Applications like shopping carts, user authentication, and personalized experiences depend on session tracking to maintain state.
Q4. Discuss the role of cookies in session management.
Answer: Cookies act as a client-side storage mechanism for session identifiers. When a user first visits, the server generates a unique Session ID and sends it as a cookie. The browser stores it and sends it back with subsequent requests. The server reads the cookie, retrieves the associated session data, and thus recognizes the returning user.
Q5. How do persistent cookies differ from non-persistent cookies?
Answer: Non-Persistent (Session) Cookies: Stored in browser memory, no expiry date, deleted when browser closes. Used for short-term session tracking. Persistent Cookies: Stored on hard drive, have a set expiry date (Max-Age), survive browser restarts. Used for "Remember Me" functionality and long-term preferences.
